Detailed checklist per organ. Use during peer review before committing READMEs.

## ORGAN-I (Theory) Validation Checklist

**For each of 10 theory repos:**

### Content Completeness
- [ ] Problem statement (why does this exist?)
- [ ] 3-5 core concepts explained clearly
- [ ] Related work cited (academic/artistic precedent)
- [ ] Installation instructions (if code)
- [ ] 2+ working examples
- [ ] Downstream implementations linked
- [ ] Roadmap or future directions
- [ ] Cross-references to other theory repos

### Accuracy & Clarity
- [ ] No jargon without explanation
- [ ] All examples actually work (tested)
- [ ] All links valid (no 404s)
- [ ] Code examples match current repo code
- [ ] Formatting is consistent + readable
- [ ] No vague statements ("interesting," "novel")

### Portfolio Relevance
- [ ] Clearly explains intellectual contribution
- [ ] Shows connection to ORGAN-II art
- [ ] Demonstrates systems thinking
- [ ] Could convince grant reviewer this matters

**ORGAN-I READY FOR LAUNCH when:**
- All 10 repos pass complete checklist
- Average length: 3,000+ words (AI-generated, human-reviewed)
- All links validated
- All examples working
- AI-validated + human-reviewed

## Quality Assurance Process (AI + Human)

**For each README:**

1. AI generates draft from organ-specific template + repo context
2. AI validation pass: template compliance, link checking, cross-reference verification, word count
3. Human strategic review: accuracy, positioning, portfolio language, repo-specific voice
4. Human revises AI output for project-specific accuracy
5. Final human sign-off
6. Commit to main branch

**Red flags during review:**
- Generic AI phrasing not customized to specific repo
- Hallucinated code examples or cross-references
- Broken links (internal or external)
- Missing cross-references
- Outdated information
- Boilerplate problem statements (not repo-specific)
- Less than minimum word count

**If red flags found:** Send back for revision. Don't commit incomplete work.
